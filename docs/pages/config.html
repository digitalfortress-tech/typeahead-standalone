Configuration

Typeahead-standalone.js provides several configuration options to fine-tune and customize autocompletion. We list all available config options below. Of these, only the first two are required whereas the rest are optional that use sensible default values.

1. üî§ input (Required)

This is the DOM input element typeahead will attach itself to. It is a required property and an error is thrown if it is missing

Javascript HTML
JS 
typeahead({
  input: document.querySelector('.myInput'),  // required
})
HTML 
<input id="typeahead" placeholder="myInput">

2. üîå source (Required)

This is the source of data from which suggestions are calculated. There are 3 types of source that can be used - local, remote and prefetch. While using all of them is not mandatory, using atleast one of them is! You may even use a combination of 2 or more source types.

Local Remote Prefetch Combination

The local data source is used when you want to provide suggestions from a local source like a variable.

JS 
// Local data source
const superheroes = ['Superman', 'Flash', 'Captain America'];

typeahead({
  input: document.querySelector('.myInput'),
  source: {
    local: superheroes
  }
});

The remote data source is used when you want to interrogate a remote endpoint to fetch data.

JS 
typeahead({
  input: document.querySelector('.myInput'),
  source: {
    remote: {
      url: 'https://remoteapi.com/%QUERY', // OR "url: (inputQuery: string) => `https://remoteapi.com/${inputQuery}`"
      wildcard: '%QUERY',
      debounce: 300,      // optional, default => 200ms
      requestOptions: {}  // optional, default => undefined
    },
  }
});

The Fetch API is used to make http requests.

Properties
  1. url (*Required) - contains the URL to which http requests are made to retrieve the suggestions. The url can be a string or a callback. When it is used as a callback, it receives the input query as a parameter and should return a string.
  2. wildcard (*Required when url is a string) - The string that is replaced with the user's input while executing the http request.
  3. debounce - A number representing the number of milliseconds to wait before executing a request to the remote endpoint. Defaults to 200ms.
  4. requestOptions - By default, the GET method is used to make http requests. To use POST method, you can provide a requestOptions property.

The response of the http requests is cached ensuring that no multiple/unnecessary requests are made for the same input/search query. If needed, this cache can be cleared using the reset API.

The prefetch data source is used when you want to preload suggestions from a remote endpoint in advance. Preloaded suggestions get added to the search index and are ready to be displayed instantly.

JS 
typeahead({
  input: document.querySelector('.myInput'),
  source: {
    prefetch: {
      url: 'https://remoteapi.com/load-suggestions',
      when: 'onFocus',          // optional, default => 'onInit'
      done: false,              // optional, default => false
      process: (items) => void, // optional, default => undefined
      requestOptions: {}        // optional, default => undefined
    }
  }
});

The Fetch API is used to make the prefetch http request.

Properties
  1. url (*Required) - contains the URL to which the prefetch request is made to preload suggestions. The url can be a string or a callback that returns a string.
  2. when - defines when the prefetch request should occur. Defaults to onInit i.e suggestions are preloaded when typeahead is initialized. You can set it to onFocus i.e. preload suggestions only when the user focuses the input element.
  3. done - A boolean used to enable/disable the prefetch request programmatically.
  4. process(suggestions) => {} - A callback that gets executed after the prefetch request occurs. It receives the suggestions as a parameter. It can be used to process the received suggestions as desired.
  5. requestOptions - By default, the GET method is used to make http requests. To use POST method, you can provide a requestOptions property.

The source can also be a combination of any two or more source types. An example using all 3 source-types is shown below.

JS 
typeahead({
  input: document.querySelector('.myInput'),
  source: {
    local: ['Superman', 'Flash', 'Captain America'],
    prefetch: {
      url: 'https://remoteapi.com/load-suggestions',
      when: 'onFocus',          // optional, default => 'onInit'
      done: false,              // optional, default => false
      process: (items) => void, // optional, default => undefined
      requestOptions: {}        // optional, default => undefined
    },
    remote: {
      url: 'https://remoteapi.com/%QUERY',
      wildcard: '%QUERY',
      requestOptions: {}        // optional, default => undefined
    }
  }
});

Here, data from the local source gets added to the search index first. Then data from the prefetch source gets added to the search index. The remote source is used only when there are few or no matching suggestions available in the search index. The suggestions from the remote source also get added to the search index.

Note: If there are duplicate suggestions coming from multiple sources, they will be merged. See advanced source configuration for further customization.

Suggestion Data-Type

The expected format of suggestions (data) can be of 2 types:
- String[] (i.e. an array of Strings) OR
- Object[] (i.e. an array of Objects)

In the example below, for the sake of simplicity, we will only use local source but the idea is valid for all types of sources (i.e. local, remote & prefetch)

String Array Object Array

When the suggestion data is an array of Strings, no additional configuration is needed and the suggestions are calculated out of the box.

JS 
// Data source as an array of Strings
const superheroes = ['Superman', 'Flash', 'Captain America'];

typeahead({
  input: document.querySelector(".myInput"),
  source: {
    local: superheroes
  }
});

Note: Internally, the source data gets normalised for further processing. In the case of the above example, the normalised data would be

JS 
/* Normalised Data */
[
  { label: "Superman" },
  { label: "Flash" },
  // ...
]
The string array gets converted into an object array with the format label: "value". The label acts like a default key.

When the suggestion data is an array of Objects, the keys property must be set. The keys array can contain multiple keys but only the first key is used to display the text within the suggestions. (The text displayed can also be overriden by using templates).

JS 
// Data source as an array of Objects
const superheroes = [
  { code: "Superman", name: "Clark Kent"},
  { code: "Flash", name: "Barry Allen"},
  { code: "Captain America", name: "Steve Rogers"},
];

typeahead({
  input: document.querySelector(".myInput"),
  source: {
    local: superheroes,
    keys: ["code"],
  }
});

The value returned by the first key (keys[0]) is expected to be unique, but if it is not, then it is highly recommended to set the identity property detailed in section below to get correct search results.

Advanced Source Configuration

When the suggestion data is an array of objects, a few additional parameters are available for more customization. All of these advanced configurations are completely optional. For the sake of simplicity, we stick to using only the local data source in the examples below

keys groupKey identity transform

The groupKey source config option is used when you wish to group suggestions together

JS 
// Local data source
const colors = [
  { id: 1, color: "Yellow", colorCode: "YW" },
  { id: 2, color: "Green", colorCode: "GN", shade: "Greenish" },
  { id: 3, color: "Olive", colorCode: "OV", shade: "Greenish" },
  // ...
];

typeahead({
  input: document.querySelector(".myInput"),
  source: {
    local: colors,
    keys: ["color"],
    groupKey: "shade" // optional
  }
});

In the above example, the colors Green and Olive will grouped together & will appear under the label "Greenish" (i.e. value of the property defined by groupKey option) whereas the color Yellow will have no group.

The groupKey config option also supports using nested properties using dot notation. For example: groupKey: "category.title"

The keys config option accepts an array of strings (which are properties of the source object) that should be added to the search index. In other words, when you want your suggestions to be searchable by using multiple properties, you list those properties in the keys config option.

Note: Although, the keys array can contain multiple keys, by default, only the first key is used to display the text within the suggestions. (Nonetheless, this behavior can overriden by using templates)

JS 
// Local data source
const colors = [
  { id: 1, color: "Yellow", meta: { colorCode: "YW"} } ,
  { id: 2, color: "Green", meta: { colorCode: "GN" } },
  { id: 3, color: "Olive", meta: {colorCode: "OV"} },
  // ...
];

typeahead({
  input: document.querySelector(".myInput"),
  source: {
    local: colors,
    keys: ["color", "meta.colorCode"],
  }
});

Given the data-source as seen in the example above, what if we wanted to search colors by another property(colorCode) and not just by its first key(i.e. color) ? By simply setting keys: ["color", "meta.colorCode"], we add the values defined in the colorCode property to the search index. Thereafter, if we search for "YW" (case-insensitive), the suggestion "Yellow" pops up as expected.

The identity() function is used to determine uniqueness of each suggestion. It receives the suggestion as a parameter and must return a string unique to the given suggestion. This is an optional property and it defaults to returning the value associated with the first key from the keys array i.e.keys[0]. However, the default value might not work everytime. For example, consider the following code -

JS 
// Local data source
const songs = [
  { title: "God is Good", artist: "Don Moen" },
  { title: "God is Good", artist: "Paul Wilbur" },
  { title: "God is Good", artist: "Micheal Smith" },
  { title: "El Shaddai", artist: "Amy Grant" },
  // ...
];

typeahead({
  input: document.querySelector(".myInput"),
  source: {
    local: songs,
    keys: ["title"],
    // identity: (suggestion) => `${suggestion.title}${suggestion.artist}`
  }
});

Given the data-source and the keys as seen in the example above, if you search for "God", you will find only 1 suggestion displayed instead of 3. This is because there are 3 songs with the exact same title (keys[0]) property. The keys[0] property which is expected to be unique, is not unique! In real world applications, such a scenario is very well possible. To handle such collisions, we use the identity() callback which returns a unique string.

In this case, we know that even though the song title's are the same, the artists aren't. So we simply return a string which concatenates the title & the artist ensuring the returned strings will always be unique. i.e.

identity(suggestion) => `${suggestion.title}${suggestion.artist}`

Ideally, if there was a unique id property available within the songs data, you could have simply returned the id (instead of concatenating 2 properties) from the identity callback to resolve the issue.

An optional transform() function is available when you need to modify the http response while using the remote/prefetch source types. This is particularly useful in situations where you are consuming a public API & over which you have no control, but you need to adapt its response as per your requirements. The transform callback is executed before the response gets processed by typeahead.

JS 
// lets assume the remote endpoint returns data in the format
[
  { id: 1, capital: "Paris", country: "France" },
  { id: 2, capital: "London", country: "UK" },
  { id: 3, capital: "Washington DC", country: "USA" },
  // ...
]

typeahead({
  input: document.querySelector('.myInput'),
  source: {
    remote: {
      url: 'https://remoteapi.com/%QUERY',
      wildcard: '%QUERY',
    },
    keys: ["place"],
    transform: (data) => {  // optional
      // transforms and returns data in the format
      // [{ id: 1, place: "Paris, France" }, ...]
      return data.map(item => ({
        id: item.id,
        place: `${item.capital}, ${item.country}`
      }));
    }
  }
});

Here, we have used the transform callback to modify the original remote response and return the modified response that contains a new property place. We are now free to use this new property anywhere we like. (we used it in the keys array).

The transform callback can additionaly be used to - flatten an array/object, add/remove properties to/from the response object, change the response format to match with a custom schema and so on...

3. üìè minLength

The minLength (Type: integer) defines the minimum number of characters required to be entered in order to trigger the calculation of suggestions. Its default value is 1.

JS 
typeahead({
  input: document.querySelector(".myInput"),
  source: {
    local: ['Superman', 'Batman', 'Flash', 'Aquaman', 'Wonder woman'],
  },
  minLength: 2, // default - 1
});

As per the example, autocompletion will be triggered only after a minimum of 2 characters have been entered.

4. ‚èπ limit

The limit (integer) config option specifies the maximum number of suggestions that should be displayed. By default 5 suggestions are shown.

JS 
typeahead({
  input: document.querySelector(".myInput"),
  source: {
    local: ['Brown', 'Black', 'Blue', 'Beige', 'Blond', 'Bronze', 'Burgundy'],
  },
  limit: 3, // default - 5
});

Going by the above snippet, a maximum of 3 suggestions will be displayed even if there are more suggestions available.

5. highlight

The highlight (Type: Boolean) config option causes the matching letters between the search query and the available suggestions to be highlighted. When enabled, the basic styling makes the matched letters bold. A class tt-highlight is added to facilitate further styling.

JS 
typeahead({
  input: document.querySelector(".myInput"),
  source: {
    local: ['Superman', 'Batman', 'Flash', 'Aquaman', 'Wonder woman'],
  },
  // highlight: false, // default - true
});

By default, this option is set to true, i.e. matching letters are highlighted. Set to false to disable this behaviour.

6. üí° hint

The hint (Type: Boolean) config option updates the input's placeholder to be equal to the first matched suggestion. Pressing the Tab key will set the input's value to the placeholder's current value.

JS 
typeahead({
  input: document.querySelector(".myInput"),
  source: {
    local: ['Superman', 'Batman', 'Flash', 'Aquaman', 'Wonder woman'],
  },
  hint: false // default - true
});

By default, the hint config option is set to true. When you set it to false, the element used to display the hint will not be rendered in the DOM. After disabling the hint, depending on your websites color theme, you may have to set a background color for your input element. (For example - .tt-input { background: #fff; }).

A class tt-hint is added to facilitate/override styling of the hint element.

7. autoSelect

The autoSelect (Type: Boolean) config option, when set to true, simply pre-selects the first displayed suggestion. Pressing the Enter key will fill the input box with the selected suggestion.

JS 
typeahead({
  input: document.querySelector(".myInput"),
  source: {
    local: ['USA', 'UK', 'UAE', 'France', 'Finland'],
  },
  autoSelect: true, // default - false
});

By default, autoSelect is set to false. The selected suggestion has a blue background by default but a class tt-selected permits further styling.

8. display

The display config option is a callback that must return a string. It receives a parameter - item i.e. the suggestion which should be used to generate the string to return. It also receives an optional 2nd parameter - event which is a Mouse/Keyboard event that can be used to track user interaction or for analytics.

The returned string is used to set the input placeholder's value (i.e. the hint). Upon selecting an item, the returned string will be set as the input's value too. Lastly, the returned string is also used to set appropriate aria-labels for each suggestion. 

<!-- Include assets -->
<link rel="stylesheet" href="https://cdn.jsdelivr.net/npm/typeahead-standalone/dist/basic.css" />
<script src="https://cdn.jsdelivr.net/npm/typeahead-standalone"></script>

<!-- input element -->
<input type="search" id="typeahead-display-eg" autocomplete="off" placeholder="Search Places...">

const places = [
  { name: 'Paris', country: 'France'},
  { name: 'London', country: 'UK'},
  { name: 'Moscow', country: 'Russia'},
  // ...
]

typeahead({
  input: document.getElementById('typeahead-display-eg'),
  source: {
    local: places,
    keys: ['name'],
  },
  display: (item, event) => {
    if (event) {
      // do something
      // alert("event type is - " + event.type);
    }

    return `${item.name}, ${item.country}`;
  }
});

The display() callback is optional and by default, it returns the string representation (usually the value of the first key keys[0]) of the selected item.

In the above input box, if you try searching for Paris, you will see the placeholder value to be "Paris, France", which is set by the display callback. Also, if you select the displayed suggestion, either by mouse click/touch or via the keyboard, the input's value will similarly be set to "Paris, France".

9. ‚ùé preventSubmit

The preventSubmit (Type: Boolean) config option is used to prevent the default submit action when the input is used inside a form element. It defaults to false

HTML Javascript
JS 
typeahead({
  input: document.querySelector("#typeahead"),
  source: {
    local: ['Superman', 'Batman', 'Flash', 'Aquaman', 'Wonder woman'],
  },
  preventSubmit: true,  // default - false
});
HTML 
<body>
  <form action="#" method="get" >
    <input type="search" id="typeahead" placeholder="Search...">
  </form>
</body>

When the input element is used inside a form element, the default action on hitting the Enter key is to submit the form. The preventSubmit flag can be used to prevent this default behavior. To later submit the form manually, you can use the onSubmit callback config option.

10. onSubmit

The onSubmit config option is a callback that gets triggered on hitting the Enter key. It receives the keyboard event as the first parameter and the selectedSuggestion as the second parameter. (i.e. only if a suggestion was selected, otherwise the 2nd parameter is undefined.)

The main utility of this callback handler is to submit the form manually especially when the input element is being used outside of a form element. 

<!-- Include assets -->
<link rel="stylesheet" href="https://cdn.jsdelivr.net/npm/typeahead-standalone/dist/basic.css" />
<script src="https://cdn.jsdelivr.net/npm/typeahead-standalone"></script>

<!-- input element -->
<input type="search" id="typeahead-onsubmit-eg" placeholder="Search Countries..." autocomplete="off">

typeahead({
  input: document.getElementById('typeahead-onsubmit-eg'),
  source: {
    local: ['USA', 'UK', 'UAE', 'France', 'Finland'],
    keys: ['name'],
  },
  onSubmit: (e, selectedSuggestion) => {
    const query = e.target.value;
    alert(`input -> ${query}, eventType -> ${e.type}`);

    if (selectedSuggestion) {
      alert('Selected suggestion - ' + JSON.stringify(selectedSuggestion));
    }
  }
});

If you try to search for "Fr", and hit the Enter key, the onSubmit callback should be triggered. You will see a single alert message. But if you search for "Fr", and then select the first suggestion (for example by pressing the down arrow key) and then hit Enter, you should see 2 alert messages as expected.

12. classNames

The classNames (Type: Object) config option simply allows you to replace the default class names as per your choice. The default class names used within typeahead are as follows:

JS 
const classNames = {
  wrapper: 'typeahead-standalone', /* main container element */
  input: 'tt-input',
  hint: 'tt-hint',
  highlight: 'tt-highlight',
  list: 'tt-list', /* container element for suggestions */
  hide: 'tt-hide',
  show: 'tt-show',
  selected: 'tt-selected',
  /* classes used within templates */
  header: 'tt-header',
  footer: 'tt-footer',
  loader: 'tt-loader',
  suggestion: 'tt-suggestion',
  group: 'tt-group',
  empty: 'tt-empty',
  notFound: 'tt-notFound',
}

/*
As an example, if you want to use a different class name for
the input element, you can initialize typeahead like this -
*/

typeahead({
  input: document.querySelector(".myInput"),
  source: {
    // ...
  },
  classNames: {
    wrapper: 'typeahead-standalone my-custom-container-class',
    input: 'my-custom-input-class' // this class is used instead of tt-input
  }
});

Note: When you use a custom class name, it replaces the default class name and hence all default styles associated with that class will disappear.

An exception to this is the wrapper class which can accept space-separated classnames. If you are using the default styles provided with this library and wish to override the wrapper class, you must also include typeahead-standalone as one of the classnames (as seen in the code snippet above). This is because the default styles are applied using this root classname.

13. ü™ô tokenizer

The tokenizer config option is a function is used to split the search query and the search data by a given character(s). This function is useful when you wish to search hypenated-words or words with a certain prefix/suffix. By default, words are split by space characters (new line, tab, spaces)

JS 
typeahead({
  input: document.querySelector(".myInput"),
  source: {
    // ...
  },
  tokenizer: (words) => {
    return words.split(/-/); // splits hypenated words
    // return words.split(/\s+/); // splits by space characters (default implementation)
    // return words.split(/\s+|-/); // splits by space characters + hyphenated words
  }
});

Imagine that your search index contains the hyphenated word "world-famous". If the default implementation of tokenizer is used, then searching for "famous" results in no results. However, if the tokenizer is set to split hypenated-words, (as shown in the above code snippet), searching for "famous" will display "world-famous" in the search results as expected.

14. üìú listScrollOptions

The listScrollOptions config option is used to have fine control over the scroll behaviour for a large list of suggestions that needs scrolling. These options are passed to the scrollIntoView() function.

JS 
typeahead({
  input: document.querySelector(".myInput"),
  source: {
    // ...
  },
  listScrollOptions: {
    block: 'center',   // default -> block: 'nearest'
    // inline: 'nearest'  // default is the same
    // behaviour: 'smooth', // default -> behaviour: 'auto',
  }
});

Using this config parameter is completely optional. The default values are
{ block: 'nearest', inline: 'nearest', behaviour: 'auto' }

15. üéØ retainFocus

The retainFocus config option is useful to control the focus on pressing the Tab key when the list of suggestions is open. When enabled (this is the default behaviour), pressing the Tab key will select the first suggestion & the focus will return to the input field. By pressing the Tab key again, the focus will move to the next element on the page. When this option is set to false, pressing the Tab key will select the first suggestion and immediately move the focus to the next element on the page.

JS 
typeahead({
  input: document.querySelector(".myInput"),
  source: {
    // ...
  },
  retainFocus: false,   // default -> true
});

15. ü™ù hooks

The hooks config option is useful to execute arbitrary code at specific moments in typeahead's lifecycle. Currently, there is 1 hook available - updateHits which is am async function that receives 1 parameter resultSet containing 3 properties:

  • - items (the suggestions returned by the search index)
  • - query (the users input query),
  • - count (the total number of suggestions found in the search index).
This hook is executed before the suggestions are displayed to the user and allows you to override the results returned by the search index in any way that you'd like. You can add, remove, modify, filter, sort suggestions and being an async function, you can also make AJAX requests to fetch new results if needed.

JS 
typeahead({
  input: document.querySelector(".myInput"),
  source: {
    // ...
  },
  hooks: {
    updateHits: (resultSet) => {
      resultSet.items.push({name: "new suggestion"}); // add new suggestion
      // resultSet.items.filter( ... ); // filter the suggestions
      // resultSet.items.sort( ... );  // custom sort the suggestions

      // you can also make an AJAX request to fetch results
      // const response = await fetch('https://example.com', { method: 'GET'});
      // const text = await response.text();
      // resultSet.items = text && JSON.parse(text);

      return resultSet; // you must return the resultSet
    },
  }
});
